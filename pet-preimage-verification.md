# Verifying halt-bridge preimages via polkadot-ecosystem-tests

Proposal for pre-alignment, addressing the Fellowship feedback: the governance tool
(app.snowbridge.network/governance) is a big ops improvement, but its output cannot be
blindly trusted because the tool and the `@snowbridge/api` npm package can be tampered
with outside Fellowship control. The generated preimage still needs a live audit during
an incident, which is exactly when nobody has time to decode SCALE bytes.

## Idea

Move the audit from incident time to review time. Commit the canonical halt and resume
preimage hex into [polkadot-ecosystem-tests](https://github.com/open-web3-stack/polkadot-ecosystem-tests)
(PET), where:

1. The hex is **version-controlled and review-gated**. Changing it requires a PR that
   maintainers (and any watching Fellow) review as a decoded diff, calmly, before any
   incident.
2. A PET test **executes that exact hex** on Chopsticks forks of live Asset Hub and
   Bridge Hub and asserts the bridge actually halts (events + storage). PET's
   `update-known-good` cron re-runs this every 6 hours against fresh chain state, so the
   reference cannot silently go stale: a runtime upgrade that breaks the call fails the
   test and pings the per-chain notification issue.

During an incident, verification collapses to one comparison: the preimage hash shown by
the tool must equal the hash of the reference hex on PET `master`. If they match, the
preimage is byte-for-byte the one that was reviewed and is continuously proven to work.
A Fellow whitelisting `whitelist.whitelistCall(hash)` checks that hash against PET's
GitHub directly, not against anything Snowbridge-hosted.

## What exists already

- `buildHaltBridgePreimage` / `buildResumeBridgePreimage` in
  `web/packages/api/src/governance/halt_bridge.ts` produce the preimage (forceBatch of
  `polkadotXcm.send` Transacts to Bridge Hub + Asset Hub calls + fee storage writes).
- `web/packages/operations/src/halt_bridge_simulation.ts` already runs the full
  halt/resume cycle on Chopsticks forks with event and storage assertions. The PET test
  is largely a port of these assertions into PET conventions.
- PET has all the machinery: `bridgeHubPolkadot` and `assetHubPolkadot` chain configs,
  `setupNetworks`, `scheduleLookupCallWithOrigin` (Root-origin dispatch via scheduler
  agenda injection), and precedent for the full whitelist pipeline in
  `packages/shared/src/upgrade.ts` (`authorizeUpgradeViaCollectives`).
- PET has **no Snowbridge tests today**; this would be net-new but only touches
  `packages/shared/src/` and `packages/polkadot/src/` (both chains are already
  configured, with CI ports, endpoints and notification issues in place).

## Blocker that had to be fixed first: preimage bytes were not runtime-stable (DONE)

`wrapBridgeHubCallsInXcmSend` queried `transactionPaymentCallApi.queryCallInfo` live and
embedded `2x queried weight` as each Transact's `fallbackMaxWeight`. Benchmarked weights
change on runtime upgrades, so the generated bytes drifted even when nothing semantically
changed, which would break the byte-for-byte comparison spuriously. A red mismatch during
a real incident is the worst possible failure mode.

**Fix shipped:** dropped the live weight query and hardcoded constant fallback weights
(`60_000_000_000` refTime / `1_048_576` proofSize). `fallbackMaxWeight` is only consulted
when the destination runtime cannot weigh the call itself, which modern BridgeHub always
can, so the value only needs to comfortably exceed any `setOperatingMode` call. The
preimage is now deterministic for a given runtime version. Remaining drift sources
(pallet index changes, XCM version bumps, call encoding changes) also break execution,
which the PET cron catches.

This diverges from the Rust `control/preimage` CLI's weight field. Decision (Clara):
accept the divergence in the TS SDK only, the TS SDK is the canonical halt source per the
emergency docs and the Rust CLI is not wired into a parity test for the halt preimage.

The frontend default lever selection and the reference are generated from one shared
constant: `FULL_HALT_OPTIONS` / `FULL_RESUME_OPTIONS` are now exported from the SDK for
use in both `HaltBridgeForm` defaults and the reference generation script.

## Pieces

### 1. Reference file in PET (the trust anchor)

JSON rather than TS so the frontend and the drift check can consume it without parsing:

`packages/shared/src/snowbridge/referencePreimages.json`

```json
{
  "generatedWith": "@snowbridge/api 1.1.9",
  "generatedAt": "2026-06-10",
  "assetHubRuntime": { "specName": "asset-hub-polkadot", "specVersion": 1009000 },
  "bridgeHubRuntime": { "specName": "bridge-hub-polkadot", "specVersion": 1009000 },
  "halt": {
    "hash": "0x...",
    "callData": "0x..."
  },
  "resume": {
    "hash": "0x...",
    "callData": "0x..."
  }
}
```

Generated by a new script `web/packages/operations/src/generate_reference_preimages.ts`
(builds both preimages with the SDK against live chains, prints the JSON). The PET PR
description includes the polkadot.js decode link for each hex so reviewers audit the
decoded call, not raw bytes.

### 2. PET test suite (draft, to be adapted to PET conventions)

`packages/shared/src/snowbridgeGovernance.ts`, registered from
`packages/polkadot/src/assetHubPolkadot.bridgeHubPolkadot.snowbridgeGovernance.e2e.test.ts`
(the two-chain filename convention matters for PET's failure routing).

```ts
import { blake2AsHex } from '@polkadot/util-crypto'
import reference from './snowbridge/referencePreimages.json'

// For each of { halt, resume }:
const { hash, callData } = reference.halt

// 0. Integrity: the committed hash matches the committed bytes.
expect(blake2AsHex(callData, 256)).toBe(hash)

const [assetHub, bridgeHub] = await setupNetworks(assetHubPolkadot, bridgeHubPolkadot)

// 1. Note the preimage (signed by a funded dev account) and dispatch it with Root
//    via scheduler agenda injection, the same way PET's upgrade suites do.
await assetHub.api.tx.preimage.notePreimage(callData).signAndSend(alice)
await assetHub.dev.newBlock()
const len = (callData.length - 2) / 2
await scheduleLookupCallWithOrigin(
  assetHub,
  { hash, len },
  { system: 'Root' },
  assetHubPolkadot.properties.schedulerBlockProvider,
)
await assetHub.dev.newBlock()

// 2. Asset Hub effects.
//    - scheduler.Dispatched ok, utility.BatchCompleted, polkadotXcm.Sent
//    - snowbridgeSystemFrontend operating mode == Halted (halt) / Normal (resume)
//    - fee storage keys == u128::MAX (halt) / prod values (resume)

// 3. Bridge Hub effects after the XCM is processed.
await bridgeHub.dev.newBlock()
//    - messageQueue.Processed success (ExpectTransactStatus did not trap)
//    - ethereumSystem.operatingMode == RejectingOutboundMessages / Normal
//    - ethereumSystemV2, inbound queues V1/V2, outbound queue, beacon client
//      operating modes == Halted / Normal

// 4. Functional check while halted: a toEthereum transfer attempt on Asset Hub
//    is rejected; after the resume preimage runs, it is accepted again.
```

v1 dispatches the raw preimage with Root, which is what the whitelist pallet does after
a referendum passes. A follow-up can run the faithful full pipeline (note wrapped
preimage, Fellowship whitelist via XCM from Collectives, `dispatchWhitelistedCallWithPreimage`),
modeled on PET's `authorizeUpgradeViaCollectives`, but that machinery is generic and
already covered by PET's upgrade tests, so it should not block v1.

Note: PET's helpers already handle the Chopsticks scheduler quirks (agenda injection,
`incompleteSince`, NonLocal block provider post-AHM), so we do not re-solve those.

### 3. Frontend comparison (convenience, not the trust anchor)

After generating, `HaltBridgeForm` fetches
`https://raw.githubusercontent.com/open-web3-stack/polkadot-ecosystem-tests/master/packages/shared/src/snowbridge/referencePreimages.json`
and compares `result.callData`:

- Default lever selection + bytes match: green banner "Matches the Fellowship-reviewed
  reference preimage" with a link to the file on PET master.
- Default selection + mismatch: prominent red warning (stale reference or tampering;
  either way, stop and verify manually).
- Custom lever selection: neutral notice that custom configurations cannot be checked
  against the reference; fall back to the decode link and KeyHasher.

The docs must state explicitly that this badge is a convenience: a compromised frontend
can fake it. The authoritative check is the human comparing the displayed hash against
PET's GitHub. That check is two strings, doable in seconds on a phone.

### 4. Drift check in Snowbridge CI

Weight or encoding drift must surface before an incident, not during one. New scheduled
GitHub Actions job (pattern already exists in `opengov_submission_check.ts`):
`web/packages/operations/src/pet_reference_check.ts` builds both preimages with the SDK
against live Asset Hub / Bridge Hub and diffs them against PET master's JSON. On
mismatch it alerts (Slack #snowbridge-security), and the team regenerates and PRs the
PET update. Runtime upgrades on AH/BH are the expected trigger.

### 5. Docs update

`docs/resources/emergency-procedures.md`, in the "Producing the preimage" and
"Submitting" steps: add the one-line verification instruction (compare the hash against
PET master, with the URL), and add it to the Fellowship escalation message template so
the whitelisting Fellow gets the reference link without asking.

## Update lifecycle

1. AH/BH runtime upgrade lands (or halt scope changes in the SDK).
2. Drift-check CI alerts, or the PET cron fails the test and comments on the
   notification issue.
3. Team runs `generate_reference_preimages.ts`, opens a PET PR with the decode links.
4. Review of the decoded diff is the audit. Merge restores green everywhere.
